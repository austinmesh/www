<hgroup>
  <h1>Getting Started with MeshCore</h1>
  <p>This page will serve well as a <b>starting point</b> for users who are familiar with Mesh networking, specifically Meshtastic, and are either looking to try, or fully switch to MeshCore. For those seeking more detail, check out a <a href="/about/meshcore-vs-meshtastic/">full comparison between MeshCore and Meshtastic</a>.</p>
</hgroup>

<section id="tldr">
  <h2>TL;DR</h2>
  <ul>
    <li>Works on (almost) all the same hardware as Meshtastic.</li>
    <li>Flash devices in-browser: <a href="https://flasher.meshcore.io/" rel="nofollow noopener">https://flasher.meshcore.io/</a></li>
    <li>Pick a role:<ul><li><strong>Companion</strong>: Bluetooth; pairs with your phone.</li><li><strong>Repeater</strong>: No Bluetooth; repeats only. Configure before deployment (use <em>Repeater Setup</em> in the flasher after flashing).</li></ul></li>
    <li>Use the <strong>US Recommended</strong> preset.</li>
  </ul>
</section>

<section id="setup">
  <h2>Setup Walkthrough</h2>

  <h3>1) Choose a Role: Companion vs Repeater</h3>
  <p><strong>Companion</strong> is your phone-connected node. <strong>Repeater</strong> is a headless coverage extender that must be configured before mounting.</p>

  <h3>2) Flash the Firmware</h3>
  <ol>
    <li>Open <a href="https://flasher.meshcore.io/" rel="nofollow noopener">flasher.meshcore.io</a>.</li>
    <li>Connect your device via USB and select the correct port/board.</li>
    <li>Select the role</li>
    <li>Enter DFU</li>
    <li>Erase the device</li>
    <li><strong>Flash</strong> the role firmware.</li>
    <li>For repeaters, click "Repeater Setup" (see below).</li>
  </ol>

  <h3>3) Configure</h3>
  <h4>Companion</h4>
  <ol>
    <li>Pair with the phone app over Bluetooth.</li>
    <li>Set <strong>Region/Preset = US Recommended</strong>.</li>
    <li>Set position if desired.</li>
    <li>Send a <b>Flood Routed Advert</b> (signal/wave icon at top of app).</li>
    <li>Remember, you may not see other contacts. Unlike Meshtastic, MeshCore only discovers other nodes when they send an advert. Patience is key!</li>
    <li><strong>optional but recommended:</strong> turn on 2-byte path size under <strong>Experimental Settings</strong>. (<a href="#2-byte-paths">Learn more about Meshcore 2-byte routing</a>)</li>
    <li>Send a test message. If you see "Heard &#123;x&#125; repeat(s)" then you are connected! If you only see "Sent", try moving to a higher location with fewer obstructions.</li>
  </ol>
  <h4>Repeater</h4>
  <ol>
    <li>After flashing, in the flasher click <strong>Repeater Setup</strong>.</li>
    <li>Set <strong>US Recommended</strong>.</li>
    <li>Set the name and position as desired. Positions are exact.</li>
    <li>Set the guest password and admin password.</li>
    <li><strong>Optional but recommended:</strong> turn on 2-byte path mode via console with command `set path.hash.mode 1` (<a href="#2-byte-paths">learn more about Meshcore 2-byte routing</a>).</li>
    <li>Reboot</li>
    <li>Send advert, make sure it shows up to your companion node</li>
    <li>From your companion node<ol><li>Log in (tap on the contact)</li><li>Go to Settings &gt; Sync Clock</li><li>Reboot</li></ol></li>
    <li>Deploy/mount the repeater.</li>
  </ol>
</section>

<section id="2-byte-paths">
  <h2>MeshCore 2-byte vs 1-byte paths</h2>
  <p>Every node(radio) on the network gets a nickname. In 1-byte mode that's something like `a1`. In 2-byte mode that's `a1b2`. 1-byte conflicts at 256 unique nicknames while 2-byte has 65,536.</p>
  <h3>Explaining key-prefixes</h3>
  <p>Every node in the mesh has a Public Key. The first characters of that key are treated as the address so other nodes know how to route messages. With 1-byte paths(the first 2 characters of the public key), there are only 256 possible nicknames to go around, so as the network grows, more and more nodes end up sharing the same one. When that happens, messages can get handed off to the wrong node and routing breaks down.</p>
  <p>2-byte paths use the first 4 characters of the public key, allowing 65,536 possibilities instead of 256. More room, fewer mix-ups, much healthier mesh as it grows.</p>
  <p><strong>The catch(es)</strong>: every repeater between you and the person you're messaging has to understand 2-byte nicknames, or it'll drop the message. That means firmware upgrades. It also reduces the max number of hops from 64 to 32.</p>
  <figure>
    <Image src={routingDiagram} alt="Diagram showing how 1 and 2 byte packets are repeated through a meshcore network with repeaters on different firmware versions" title="only repeaters with firmware at or above 1.14.1 repeat 2-byte packets" />
    <figcaption>only repeaters with firmware at or above 1.14.1 repeat 2-byte packets</figcaption>
  </figure>
  <h3>Requirements</h3>
  <ul>
    <li><strong>Repeaters:</strong> firmware <strong>1.14 or later</strong>. Older repeaters can't forward 2-byte paths.</li>
    <li><strong>Companions:</strong> firmware <strong>1.14 or later</strong>, plus the latest phone app (the toggle lives under Experimental Settings).</li>
  </ul>
  <p><em>If any repeater in the path is on older firmware, 2-byte traffic won't make it through</em>. Plan upgrades accordingly.</p>
  <h3>Turning it on</h3>
  <h4>Repeaters</h4>
  <ol>
    <li>Upgrade to at least 1.14</li>
    <li>Connect to the repeater's console and run: <pre><code>set path.hash.mode 1</code></pre></li>
  </ol>
  <h4>Companions</h4>
  <p>Companions need an extra step to actually <em>send</em> messages using 2-byte paths:</p>
  <ol>
    <li>Confirm the companion is on firmware 1.14 or later.</li>
    <li>Open the phone app.</li>
    <li>Go to <strong>Experimental Settings</strong>.</li>
    <li>Set <strong>Default Path-Hash Size</strong> to 2-byte (max 32 hops).</li>
  </ol>
  <p>Once that's on, messages from the companion will use 2-byte paths when available.</p>
</section>
